# Phase 3: v0.3 — API backend — PLAN

**Wave:** 1 (sequential: api-setup → auth/steam/players in parallel → matches-api validation)

## Objective

Complete the API layer: authentication, Steam integration, player management, and match webhook handling. Enable the portal to communicate with game servers and store player data.

## Scope

This phase delivers the REST API that powers the web portal and integrates with game servers:

1. **api-setup** — Foundation (Node.js + Express + TypeScript + Knex, systemd service)
2. **auth-api** — Authentication (Google OAuth, email/password, JWT refresh tokens)
3. **steam-integration** — Steam OpenID linking, public player endpoint
4. **players-api** — Player CRUD, public profiles, ban management
5. **matches-api** — Match webhook ingestion (MatchZy/Get5), stats persistence

## Verification Criteria

- [ ] All 5 features have Validate gate approved (includes E2E testing)
- [ ] `npm run build` succeeds (TypeScript compilation)

## Implementation Notes

### Feature Dependencies

```
api-setup → (auth-api, steam-integration, players-api) [parallel] → matches-api [sequential]
```

- **api-setup** must complete first (project structure, dependencies, config)
- **auth-api, steam-integration, players-api** can be developed in parallel
- **matches-api** depends on all 4 above (uses all CRUD endpoints, auth middleware, config patterns)

### Key Integration Points

- **auth-api** provides `POST /auth/register`, `POST /auth/login`, `POST /auth/refresh`
  - Sets `access_token` (JWT, short-lived) and `refresh_token` (long-lived, rotated on refresh)
  - `authMiddleware` validates JWTs on protected routes

- **steam-integration** integrates with auth-api
  - Requires authenticated user to link Steam (`POST /auth/steam/link`)
  - Public endpoint `GET /api/player/:steamid` returns player profile without auth

- **players-api** extends auth-api
  - `GET /api/players` — public list (paginated, no sensitive fields)
  - `GET /api/players/:id` — public profile
  - `PATCH /api/players/me` — authenticated user's own profile
  - Admin role required for ban operations

- **matches-api** uses all above
  - Webhook ingests MatchZy/Get5 events
  - Updates `matches` and `stats` tables
  - Calls `eloUpdateStub` (future integration point)
  - Sends Discord notifications (optional, non-blocking)
  - Public endpoints: `GET /api/matches`, `GET /api/matches/:id`, `GET /api/players/:id/matches`

### Environment Variables

Defined in `.env.example` and validated via `src/config/env.ts` (zod schema):

**From api-setup:**
- `NODE_ENV` (development/production)
- `PORT` (default 3000)
- `DATABASE_URL` (MariaDB connection string)
- `LOG_LEVEL` (debug/info/warn/error)

**From auth-api:**
- `JWT_SECRET` (min 32 chars)
- `JWT_REFRESH_SECRET` (min 32 chars)
- `GOOGLE_CLIENT_ID`, `GOOGLE_CLIENT_SECRET`
- `GOOGLE_REDIRECT_URL`

**From steam-integration:**
- `STEAM_REALM` (http://localhost:3000 in dev, domain in prod)
- `STEAM_RETURN_URL` (callback URL)
- `STEAM_STATE_SECRET` (min 32 chars)

**From matches-api:**
- `WEBHOOK_SECRET` (min 32 chars, generated by api-setup.sh)
- `DISCORD_WEBHOOK_URL` (optional, for notifications)

### Testing Strategy

- **Unit tests:** Parser validation, utility functions (no DB)
- **Integration tests:** Auth flows, endpoint behavior (uses test DB)
- **E2E tests:** Webhook payload processing (MatchZy samples, Get5 samples)

See `.specs/features/{slug}/validation.md` for detailed test cases per feature.
