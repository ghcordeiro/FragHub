Create `src/services/queueService.ts` implementing:

**State Machine** (per QUEUE-REQ-006):
- Enum `QueueState` = WAITING_PLAYERS, PLAYERS_FOUND, MAP_VOTE, IN_PROGRESS, FINISHED
- State transitions strictly enforced (e.g., can't go from FINISHED back to MAP_VOTE)
- Store state in `queue_sessions` table

**Core Functions**:

`joinQueue(userId: string, knex: Knex): Promise<{ position: number; totalInQueue: number }>`
- Validate user authenticated (caller provides userId from req.user)
- Check user has Steam linked (query users table, steam_id IS NOT NULL) per QUEUE-REQ-001
- Check user not already in active queue session (no row with state != FINISHED) — return 409 if true (QUEUE-REQ-001)
- Check user not in in-progress match (check queue_players.queue_session_id where state = IN_PROGRESS)
- Add user to current queue_sessions row (or create if none in WAITING_PLAYERS state)
- Insert into queue_players (queue_session_id, user_id, team_assignment=NULL, joined_at=NOW)
- Update queue_players.last_activity = NOW on every join
- Return position in queue (1-indexed) and total count
- Rate limit per QUEUE-NFR-004: caller enforces (express-rate-limit middleware, not in service)

`leaveQueue(userId: string, knex: Knex): Promise<void>`
- Remove queue_players row matching user_id from active session (QUEUE-REQ-002)
- Always returns 200 (idempotent per spec)
- If queue becomes empty, delete the queue_sessions row

`getQueueStatus(userId: string, knex: Knex): Promise<QueueStatusResponse>`
- Query user's current queue_players row
- If not found, return `{ state: "NOT_IN_QUEUE" }`
- If found, return state + position + totalInQueue + if state >= PLAYERS_FOUND, include team composition (both teams' user objects with elo/level)
- If state = MAP_VOTE, include `currentVetoState: { banHistory: [{banningTeam, map}...], remainingMaps: [...], currentTurn: "TEAM_A" | "TEAM_B", isCaptain: boolean }`
- If state = IN_PROGRESS, include `connectString: "ip:port"`, `mapSelected: string`
- If state = FINISHED, return `{ state: "FINISHED" }` (user still in queue_players until cleanup, but UI treats as ready to join new queue)

`balanceTeams(userIds: string[], knex: Knex): Promise<{ teamA: string[]; teamB: string[]; avgEloA: number; avgEloB: number }>`
- Called when 10 players assembled
- Fetch user objects (including elo_rating) for all userIds
- Sort by ELO descending
- Apply "snake draft" (QUEUE-REQ-004): distribute players alternately (1→A, 2→B, 3→B, 4→A, 5→A, 6→B, 7→B, 8→A, 9→A, 10→B)
- Verify ELO difference between team averages < MAX_ELO_DIFF from env (default 50)
- If diff > MAX_ELO_DIFF, re-order using greedy balancing: alternate between highest and next-lowest available until balanced
- Return teamA (array of 5 user IDs), teamB (array of 5 user IDs), plus calculated averages
- Update queue_players.team_assignment = "TEAM_A" | "TEAM_B" in database

`advanceQueueState(currentState: QueueState, nextState: QueueState, queueSessionId: string, knex: Knex, context?: {mapSelected?, connectString?}): Promise<void>`
- Validate state transition is allowed
- Update queue_sessions.state = nextState, updated_at = NOW
- If nextState = PLAYERS_FOUND, nothing else
- If nextState = MAP_VOTE, initialize veto state in-memory (remainingMaps = QUEUE_MAP_POOL from env)
- If nextState = IN_PROGRESS, fire Discord notification (via discordNotifyService.notifyMatchReady)
- If nextState = FINISHED, remove queue_players rows for this session, fire Discord notification (via discordNotifyService.notifyMatchComplete)

`checkQueueTimeouts(knex: Knex): Promise<void>`
- Background task (called every 60 seconds by routes/queue.ts timer)
- Query queue_players where queue_session.state = WAITING_PLAYERS and joined_at < NOW - QUEUE_TIMEOUT_MINUTES
- Remove timed-out players from queue
- If queue becomes empty, delete queue_sessions row
- Log INFO on each removal (QUEUE-NFR-005)

**Veto State Management** (per QUEUE-REQ-007):
- Store in-memory: vetoState = { banHistory, remainingMaps, currentTurn, vetoStartTime }
- `voteMap(userId: string, action: "ban", map: string, queueSessionId: string, knex: Knex): Promise<void>`
  - Verify user is captain (highest ELO in their team)
  - Verify it's user's team's turn
  - Remove map from remainingMaps
  - Record in banHistory
  - If remainingMaps.length = 1, auto-advance to IN_PROGRESS with selectedMap = remainingMaps[0]
  - Timeout per veto turn: 30 seconds (VETO_TIMEOUT_SECONDS env var) — if no vote received, randomly ban a map and advance turn (implement via setTimeout in memory, fire-and-forget)

**Lock/Concurrency** (per QUEUE-NFR-002):
- Node.js single-threaded guarantees atomicity of synchronous operations
- All queue modifications must be synchronous before awaiting Knex calls
- Example: read queue state, validate, construct changes, then await Knex update (no await in between)

Ensure all logging uses logger.info/warn/error per QUEUE-NFR-005.

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| Client → API Queue | Untrusted: players could try to join queue without Steam, manipulate team assignment, flood queue |
| API → Game Server RCON | Untrusted: game server responses to RCON commands could be spoofed; API validates server state after RCON |
| Game Plugins → API | Untrusted: plugins fetch public player data; network could be intercepted (mitigation: HTTPS in production) |
| API → Database | Trusted: internal network, Knex parameterized queries prevent injection |
| Discord Webhook | Semi-trusted: webhook URL is configuration secret; if leaked, attacker can spam channel but cannot retrieve data |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-05-01 | Tampering | Queue state (in-memory) | Mitigate | Store queue state in database (queue_sessions, queue_players tables); validate state transitions before advancing; log all state changes for audit |
| T-05-02 | Spoofing | Player identity in queue | Mitigate | Validate JWT on all /api/queue/* endpoints; extract userId from req.user (set by authMiddleware); rate-limit joins (1 per 30s) |
| T-05-03 | Repudiation | ELO updates | Mitigate | Log all ELO changes to elo_history table with match_id, user_id, before/after values; idempotency check prevents duplicate ELO application |
| T-05-04 | Information Disclosure | Discord webhook URL | Accept | DISCORD_WEBHOOK_URL is configuration secret; if attacker gains access to .env file, they can spam Discord but not modify data; mitigate via file permissions (600 on .env), not code-level |
| T-05-05 | Denial of Service | Queue join flooding | Mitigate | Rate limit POST /queue/join (1 per 30s per player); QUEUE_TIMEOUT_MINUTES removes inactive players; validate Steam link before adding to queue |
| T-05-06 | Denial of Service | Plugin HTTP requests | Mitigate | Plugin enforces 3-second timeout; graceful fallback (no tag) if API unreachable; plugin does not retry or queue requests |
| T-05-07 | Elevation of Privilege | Team assignment via RCON | Mitigate | RCON commands constructed server-side, not user input; game server validates team assignments before match start; no player-controlled RCON injection |
| T-05-08 | Logic bypass | Veto map selection | Mitigate | Validate captain role (highest ELO in team); enforce turn order in state machine; timeout veto turns with automatic ban if no vote; log all votes |
| T-05-09 | Integrity | Webhook secret for ELO updates | Mitigate | POST /api/matches/webhook checks X-Webhook-Secret header against WEBHOOK_SECRET env var (shared with game server); only authenticated webhooks update ELO |
| T-05-10 | Information Disclosure | ELO history query | Accept | GET /api/players/:id/elo-history requires authentication; user can only see their own history or admins can see any player; public leaderboard shows only Level (1-10), not exact ELO |

**Summary**: Phase 5 threats are primarily around queue state manipulation and denial of service. Mitigations: database persistence (not in-memory-only), JWT authentication, rate limiting, webhook secret validation, and graceful degradation for external services (Discord, plugins).
