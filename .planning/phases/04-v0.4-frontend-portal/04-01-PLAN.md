2. Template for /etc/nginx/sites-available/fraghub (parametrized):

   # FragHub Nginx configuration
   # Serves React frontend from /opt/fraghub/portal/dist
   # Proxies /api/* to Node.js backend on localhost:3000
   
   upstream fraghub_api {
       server 127.0.0.1:3000;
   }
   
   server {
       listen 80;
       server_name _;
       
       # Security headers
       add_header Strict-Transport-Security "max-age=63072000; includeSubDomains" always;

       root /opt/fraghub/portal/dist;
       index index.html;
       
       # SPA routing fallback
       location / {
           try_files $uri $uri/ /index.html;

## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| Client → Nginx | HTTP(S) requests from browser; Nginx validates and proxies |
| Browser → Frontend app | React code executed in browser context; XSS risk if user input not escaped |
| Frontend → Nginx API proxy | Requests with Authorization header; JWT in memory protects against XSS cookie theft |
| Nginx → Backend API | Reverse proxy; backend validates JWT and enforces authorization |
| Nginx → Let's Encrypt | ACME protocol for certificate renewal; DNS propagation required |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-04-01 | Spoofing | OAuth callback (/auth/callback) | Mitigate | Validate token signature on backend before storing in sessionStore; extract token from URL and immediately remove via history.replaceState to prevent token leakage in browser history |
| T-04-02 | Tampering | Login form | Mitigate | Use HTTPS (certbot-enforced); validate email/password format client-side and server-side; enforce min password length (8 chars per spec) |
| T-04-03 | Tampering | Token in URL | Mitigate | OAuth tokens appear in URL only during callback; immediately consumed and removed; never log tokens in console |
| T-04-04 | Repudiation | User edits profile | Mitigate | Backend logs PATCH requests with user ID and timestamp (audit trail); frontend confirms success before showing updated UI |
| T-04-05 | Information Disclosure | XSS attack | Mitigate | React escapes JSX by default; avoid dangerouslySetInnerHTML; sanitize any user input displayed; CSP header blocks inline scripts and limits script sources to 'self' |
| T-04-06 | Information Disclosure | Access token exposure | Mitigate | Store access token only in Zustand (memory), never in localStorage/sessionStorage; refresh token in httpOnly cookie (set by backend, inaccessible to JS); leverage HTTPS to prevent network interception |
| T-04-07 | Denial of Service | Leaderboard query | Mitigate | Backend enforces rate limiting via middleware (existing from auth-api); frontend respects pagination limit (25 per page); query parameters validated server-side |
| T-04-08 | Elevation of Privilege | Protected routes | Mitigate | ProtectedRoute component checks user.role from sessionStore; backend enforces authorization on all /api/players/me endpoints; 403 Forbidden on insufficient privileges |
| T-04-09 | Access Control | Public profile pages | Accept | /players/:id is public by design (no authentication required); backend does not expose sensitive fields (password, refresh token, email) in public player endpoints |
| T-04-10 | CSRF | Form submissions | Mitigate | All mutating requests (POST /auth/login, PATCH /api/players/me) use standard fetch with Authorization header; Nginx CSP header blocks cross-origin form submissions; backend can add CSRF tokens if needed in future |
